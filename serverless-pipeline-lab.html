What you'll build

One pipeline, three primitives

The whole system reduces to three serverless building blocks. Get each working alone, then connect them. Here's the mental model before we touch a terminal.

Cloud Run Job = run-to-completion container

Unlike a Cloud Run service (which listens on a port forever), a job runs your code and exits. No web server, no request handling — perfect for scripts and batch work. A job can fan out to thousands of parallel tasks, each seeing CLOUD_RUN_TASK_INDEX.

Pub/Sub = decoupled messaging

A publisher sends bytes to a topic; a subscription delivers them to consumers. The publisher never knows who's listening. Messages persist (7 days by default) and must be acknowledged within the ack deadline (10–600s) or they're redelivered.

Cloud Scheduler = managed cron

It doesn't run your code — it fires an HTTP request (or Pub/Sub message) on a unix-cron schedule. "Run my job every morning" becomes a timer that calls an API. Think of it as cron + an HTTP call, fully managed.

Stage 01

The simplest possible Cloud Run Job

Goal: run Python in the cloud and watch it exit
Start → end of this stepWrite main.pyAdd Procfile + requirements.txtDeploy from sourceExecute the jobContainer exits 0

A job packages your code into a container, runs it to completion, and stops. Deploying from source means no Dockerfile — Cloud Build turns your script into an image with buildpacks, pushes it to Artifact Registry, and registers the job. Start with the smallest program that proves it works.

You create three tiny files

A source deploy uses buildpacks, and the Python buildpack assumes you're deploying a web app — by default it starts gunicorn main:app. A job isn't a web app and has no app, so without an explicit entrypoint it crashes on startup. The Procfile is what tells it to simply run your script. So: main.py (your code), Procfile (the entrypoint), and requirements.txt (so Python is detected).

main.py
main.pypython 
print("Hello from Cloud Run Job!")
Procfile — no extension; this is the fix
Procfiletext 
web: python main.py
requirements.txt — can be empty; its presence makes buildpacks pick Python
requirements.txttext 
# this first job needs no third-party packages
Deploy from source, then execute

Where does the name “hello-job” come from?

You do — it's not automatic and not a default. In gcloud run jobs deploy hello-job --source ., the word right after deploy is the job's name, and you typed it. It isn't taken from the folder, from main.py, or from --source .. If you ran gcloud run jobs deploy my-job … the job would be called my-job. The job name is fixed once created (you'd delete and recreate to rename). The execution IDs like hello-job-gskgz are auto-generated — that's your job name plus a random suffix Cloud Run adds each run.

Reading the commands: pink = your value

Later steps need values that Google assigns to your project (a project number, a service URL). Those appear as pink variables like $PROJECT_NUMBER. A line just above each one captures it — e.g. PROJECT_NUMBER=$(gcloud … ) — so you can copy-paste the whole block and never hand-edit anything.

From the docs

Each execution gets a unique ID and streams to Cloud Logging the moment it starts. Add --execute-now to jobs deploy to run immediately on first deploy. Default task timeout is 10 min (max 7 days); default retries per task is 3 — that's why a failing task shows three attempts before it gives up.

Gotcha

A job must not start a web server or listen on a port — that's a service. The Procfile line web: python main.py just sets the container's start command; the word “web” is a process-type label, not a server.

Pass configuration with environment variables

Goal: see how config reaches the container
Start → end of this stepEdit main.py to read NAMERedeploy with --update-env-varsExecuteLogs show your NAME

Hard-coding values is a dead end. Environment variables let the same image behave differently per run. Read them in Python with os.getenv, and override them at execution time — this is the exact pattern an ingestion framework uses to tell one job which dataset to process.

main.py
main.pypython
import os

print("NAME:", os.getenv("NAME", "World"))
Ship the new code + the variable, then run
Cloud Shellbash
# main.py now READS the NAME variable (see above), so you must DEPLOY again to
# ship that new code — setting a variable alone won't change a container that
# doesn't read it. --update-env-vars sets NAME in the same step.
#   deploy --source .          : rebuild the image from the updated main.py
#   --update-env-vars NAME=...  : add/append the variable (keeps any existing ones)
gcloud run jobs deploy hello-job --source . --region us-central1 \
  --update-env-vars NAME=Subhajit
#» Building using Buildpacks and deploying container to Cloud Run job [hello-job]...
#» Done. Job [hello-job] has successfully been deployed.

# run it — the container now has BOTH the new code AND the NAME value
gcloud run jobs execute hello-job --region us-central1 --wait
#» Running execution... 1 / 1 complete... done
#» Execution [hello-job-spd6z] has successfully completed.

# to change ONLY the value later, no rebuild is needed (the image already reads NAME);
# the new value applies on the next execution:
#   gcloud run jobs update hello-job --update-env-vars NAME=World

--set-env-vars is destructive

It deletes every variable not in the new list. To add/modify without clobbering, use --update-env-vars; to delete specific keys, --remove-env-vars. For values containing commas, set a delimiter: "^##^K=a,b,c".

Never put secrets here

Env vars are visible to anyone with Viewer or higher. Use Secret Manager for API keys, and never set GOOGLE_APPLICATION_CREDENTIALS — attach a service account instead. Max 1,000 vars per container.

Pub/Sub on its own

Goal: send and receive exactly one message
Start → end of this stepCreate topic + subscriptionRun publisher.pyPull the subscriptionMessage received

Before connecting Pub/Sub to anything, prove the basics. Create a topic, attach a subscription, publish bytes from Python, then pull them back from the terminal. The publisher and the puller never reference each other — that decoupling is the whole point.

Create the topic & subscription
Cloud Shellbash
# a TOPIC is the channel that publishers send messages to
gcloud pubsub topics create demo-topic
#» Created topic [projects/serverless-pipeline-lab/topics/demo-topic].

# a SUBSCRIPTION is how a reader pulls messages from that topic.
#   --topic         : which topic this subscription listens to
#   --ack-deadline  : seconds you have to acknowledge before redelivery (10–600)
gcloud pubsub subscriptions create demo-sub \
  --topic=demo-topic \
  --ack-deadline=60
#» Created subscription [projects/serverless-pipeline-lab/subscriptions/demo-sub].
Publish from Python
publisher.pypython
from google.cloud import pubsub_v1

publisher = pubsub_v1.PublisherClient()
topic_path = publisher.topic_path("serverless-pipeline-lab", "demo-topic")

# data MUST be a bytestring
future = publisher.publish(topic_path, b"Hello PubSub!")
print("Message sent:", future.result())
Run it in Cloud Shell

Stage 06
06

Scheduler triggers the Cloud Run Job

Goal: your primary orchestration pattern
Start → end of this stepCreate scheduler-sa + grant run.invokerSchedule the :run call (OAuth)Job runs on cron

This is the keystone — and the single most common place people get stuck. Scheduler calls the Cloud Run Admin v2 API :run endpoint. Because the target lives on run.googleapis.com (a Google API), it needs an OAuth token, not OIDC. Give a service account roles/run.invoker and pass it via --oauth-service-account-email.

Create the invoker SA & grant the role

Schedule the job via the v2 :run endpoint (OAuth)
Cloud Shellbash
# Re-derive your values so this block stands on its own ($PROJECT_ID, $REGION, $SCHED_SA).
PROJECT_ID=$(gcloud config get-value project)
REGION=us-central1
SCHED_SA="scheduler-sa@$PROJECT_ID.iam.gserviceaccount.com"

# Schedule the JOB by POSTing to the Cloud Run Admin v2 ":run" endpoint.
#   --schedule   : "0 12 * * *" = every day at 12:00
#   --uri        : the v2 jobs:run URL, built from $PROJECT_ID and $REGION
#   --http-method: POST (":run" is triggered with POST)
#   --oauth-service-account-email : OAuth token as $SCHED_SA. OAuth — NOT --oidc-* —
#                  because the target is a *.googleapis.com Google API.
gcloud scheduler jobs create http run-hello-job \
  --location="$REGION" \
  --schedule="0 12 * * *" \
  --uri="https://run.googleapis.com/v2/projects/$PROJECT_ID/locations/$REGION/jobs/hello-job:run" \
  --http-method=POST \
  --oauth-service-account-email="$SCHED_SA"
#» Created job [run-hello-job].

OAuth, not OIDC — this is the trap

OIDC tokens fail against run.googleapis.com. Use --oauth-service-account-email (optionally with scope https://www.googleapis.com/auth/cloud-platform). --location is effectively required; confirm it in gcloud scheduler locations list.

Override env vars per fire
