01 — The mental model

One sentence to hang everything on

A change happens somewhere. Pub/Sub catches it and Eventarc routes it; Cloud Run processes it; Airflow/Composer coordinates the bigger job; the Catalog governs it and it settles into a system of record. Four colours, four jobs — that's the whole map.

INGEST · TRANSPORT

Move the message

Get the event off the source onto a durable, decoupled channel. Pub/Sub, Eventarc.

COMPUTE

Run the code

Do the work — transform, enrich, call an API — on demand, scaling to zero. Cloud Run, Cloud Run functions.

ORCHESTRATE

Coordinate the steps

Sequence many tasks in order, on a schedule, with retries and visibility. Airflow, Composer, the DAG.

GOVERN · RECORD

Make it trustworthy

Catalogue, secure and add meaning; land the authoritative copy. Knowledge Catalog, the SOR.

Google Cloud Pub/Sub is a globally distributed, asynchronous messaging service. Publishers write messages to a topic; independent subscribers read them through subscriptions — neither side knows the other exists.

The problem it solves

Without a message broker, every producer has to know every consumer, hold a connection open, and slow down if a consumer is busy. Pub/Sub breaks that coupling: it absorbs spikes, buffers durably, and lets you add or remove consumers without touching producers. It's the front door for most event-driven and streaming pipelines on GCP.

The four words to know
Topic
A named resource a publisher sends to — the stream's identity.
Subscription
A named pipe from a topic to a consumer. Push (Pub/Sub calls your endpoint) or pull (you fetch).
Message
The payload plus optional key–value attributes.
Ack
The consumer's confirmation that a message was processed; un-acked messages are redelivered.
Delivery guarantees
  • At-least-once by default — design consumers to be idempotent (the same message may arrive twice).
  • Ordered delivery is opt-in, and only within the same ordering key.
  • Exactly-once delivery is available as a subscription option.
  • Dead-letter topics catch messages that repeatedly fail, so a poison message can't block the stream.
Where it sits: Pub/Sub is the ingest layer. Common shape: producer → topic → subscription → Cloud Run / Dataflow → BigQuery. Native BigQuery and Cloud Storage subscriptions can land data with no code at all.

📰
A newspaper subscription

Publishers drop one edition into a topic; every subscriber gets their own copy, and the publisher never needs to know who's reading.

Mental model

Think of a durable buffer that decouples senders from receivers. The buffer holds messages until each consumer confirms it's done, and lets you fan one message out to many consumers (fan-out) or many producers into one stream (fan-in).

Onboarding gotchas
  • At-least-once means duplicates happen — make handlers idempotent.
  • Ordering is not global; it only holds within an ordering key.
  • A topic with no subscription drops messages — the subscription is what retains them.

Cloud Run is Google's fully managed serverless container platform: package anything as a container and Cloud Run runs it, scales it, and (for services) scales it to zero when idle.

Three resource types — know the difference
Service
Responds to HTTP(S) requests. Autoscales on traffic, supports concurrency and request timeouts up to 60 min.
Job
Runs a container to completion then stops — batch transforms, migrations, nightly reports. No HTTP listener. Tasks can run up to 7 days.
Worker pool
Non-HTTP, pull-based workloads (e.g. queue consumers). Reached GA in April 2026.
Why people like it
  • Any language, any binary — if it runs in a container, it runs here.
  • Revisions & traffic splitting — deploy a new revision, send 5% of traffic, roll forward or back.
  • Scale to zero (services) — you pay for what you use.
  • Array jobs — a job can fan out into many parallel tasks via task index.
Jobs vs services in one line: a service waits for requests; a job does a unit of work and exits. In a pipeline, a Cloud Run job is the natural home for a heavy transform step that Airflow triggers.

A food truck

The grill only fires up when an order arrives, then powers down. A job is the same kitchen booked for one catering run — it cooks the order and closes.

Mental model

Your container, scaled on demand. A service is a kitchen that opens whenever a customer walks in; a job is that kitchen hired for a single fixed catering order.

Onboarding gotchas
  • Services scale to zero → first request after idle pays a cold start.
  • Jobs don't serve traffic — don't put an HTTP server in a job.
  • Concurrency is per-instance; a misconfigured concurrency of 1 quietly multiplies your cost.

Cloud Run functions — the service formerly called Cloud Functions — runs a single function in response to an HTTP request or an event, with no container or server to manage.

What it's for

It's Functions-as-a-Service: write one handler, deploy it, and Google runs and scales it. Perfect for glue — resize an uploaded image, validate a webhook, react to a file landing in storage, transform a Pub/Sub message. You bring a function; the platform brings everything else.

How it's triggered
  • HTTP — a URL endpoint.
  • Events — via Eventarc and Pub/Sub, from 125+ Google and third-party sources (storage, Firestore, audit logs, and more).
The big idea behind the 2024 rename: Cloud Run functions now runs on the same infrastructure as Cloud Run. A function is essentially a Cloud Run service that Google builds and manages for you from your source — so it inherits Cloud Run's concurrency, scaling and networking.

Knowledge Catalog — the product that began as Dataplex — discovers, catalogues, governs and adds meaning to data across your lakes and warehouses, from one control plane.

What it does
  • Discover & catalogue — automatically finds datasets (BigQuery, Cloud Storage and more) and builds a searchable catalog of metadata.
  • Govern — apply policies, quality checks and access controls consistently, regardless of where the data physically lives.

A DAG is a Directed Acyclic Graph: nodes connected by one-way edges, where following the arrows never loops back to where you started.

Unpack the three words
Directed
Edges have a direction — A → B means "A before B."
Acyclic
No cycles — you can never follow the arrows in a loop.
Graph
A set of nodes and the edges between them.

Because there are no cycles, a DAG can always be put in a valid running order (a topological sort). That's exactly why orchestration tools model workflows as DAGs: tasks become nodes, dependencies become edges, and the engine can always work out what to run next.

Not a product. A DAG is a concept from mathematics and computer science — there's no version number, no vendor. It just shows up everywhere work has dependencies.
Where the idea comes from

The DAG predates every data tool. It's a core structure in graph theory, and you've already used it without naming it:

  • Git history — commits point to parents, never to descendants.
  • Build systems (Make, Bazel) — compile dependencies before the thing that needs them.
  • Spreadsheets — a cell's formula depends on other cells; circular references are forbidden (that's the "acyclic" rule).
  • Citations, task schedules, package managers — all DAGs.
How "DAG" shows up in Airflow

In Airflow a DAG is defined in a Python file; the tasks are Operators, Sensors or TaskFlow functions. There are two authoring styles:

  • Traditional — the DAG class / context manager, wiring tasks with >>.
  • TaskFlow API — the @dag and @task decorators, introduced in Airflow 2.0 (Dec 2020), which infer dependencies and pass data automatically.
ConventionDetail
CasingHistorically "DAG" (the acronym); Airflow 3 docs stylise it "Dag" in prose, while code keeps dag_id, @dag, the DAG class.
Import (Airflow 3)Moved to airflow.sdk — e.g. from airflow.sdk import dag.
🚇
A one-way subway map

One-way tracks and no loops. You always reach the end of the line and never circle back to the station you started at.

Mental model

Tasks as nodes, dependencies as one-way edges, never a cycle. Or: a recipe's flowchart — you can't frost before you bake, and the steps always have a valid order.

Onboarding gotchas
  • A cycle is illegal — if task A waits on B and B waits on A, the graph won't run.
  • Tasks should be idempotent, because a DAG run can retry any node.

Apache Airflow is the open-source standard for orchestrating workflows. You write pipelines as Python — "workflows as code" — and Airflow schedules, runs, retries and monitors them.

The moving parts
Scheduler
Decides what runs when, and dispatches tasks.
Executor / workers
Actually run the tasks (locally, on Celery, on Kubernetes…).
Metadata DB
Stores DAG state, runs, history.
Web UI
Visualise DAGs, inspect runs, retry, read logs.
The vocabulary
  • Operators — a unit of work (run SQL, call an API, launch a container).
  • Sensors — wait for a condition (a file appears, a time passes).
  • Hooks — reusable connections to external systems.
  • XCom — the channel tasks use to pass small bits of data.
  • Providers — a huge ecosystem of pre-built integrations (GCP, AWS, Snowflake, dbt…).

A System of Record (SOR) is the authoritative source for a given piece of data — the one place where that data's value is definitively established and maintained.

What it means

When the same data lives in several systems, the SOR is the master that arbitrates. Changes are made there first and propagate outward; every other copy is downstream. It's foundational in data architecture because it answers a deceptively hard question: when two systems disagree, which one is right?

It's domain-specific
  • An HR system is the SOR for employee identity.
  • An ERP is the SOR for financials.
  • A banking core / CIS is the SOR for an account balance.
Not a product. "System of Record" is a role a system plays, not software you buy. In Data Vault modelling the same idea is called the "record source."
The family of terms (don't mix these up)
System of engagement
User-facing, collaborative systems (CRM, mobile, social). Term coined by Geoffrey Moore in 2011. SORs are about efficiency & compliance; systems of engagement about effectiveness.
System of reference
An authoritative, curated copy used for lookup — often equated with "source of truth."
Golden record
In Master Data Management, the single consolidated, deduplicated "best" version of an entity (customer, product).
Single source of truth
The broader practice of keeping one authoritative value per data element.
Where the SOR sits in a pipeline

It's the destination end of the flow. After events are ingested, routed, processed and orchestrated, the curated, authoritative result lands in a system of record — often a BigQuery table or an operational database — with the catalog governing it and lineage recording where it came from.

  • The SOR holds the version everyone else trusts and copies from.
  • Reporting marts, caches and dashboards are copies, not the SOR.
  • Governance (Knowledge Catalog) plus lineage is what lets you prove a value's provenance back to its SOR.
🏛️
The government birth registry

Lots of systems store your birthdate — your bank, your airline, your doctor — but only one is the official truth. Everything else is a copy of the registry.

Mental model

The one place a fact is officially true. If you need to change the fact, you change it in the SOR; everywhere else eventually catches up.

Onboarding gotchas
  • Don't confuse the SOR (authoritative) with a cache or reporting copy.
  • "System of record" vs "system of engagement" is a common interview distinction — engagement is the front-of-house, record is the source of truth.

"Routing" is the job of sending each message, event or request to its correct destination. On GCP the dedicated event-routing service is Eventarc.

What Eventarc does

Eventarc listens for "something happened" across Google Cloud and delivers a standardised event to your handler. It uses the open CloudEvents format, draws from 130+ sources, and delivers to targets like Cloud Run, GKE, Workflows and internal HTTP endpoints — with Pub/Sub as the transport underneath.

  • Sources — direct events from ~90 Google services, Cloud Audit Logs, and Pub/Sub messages.
  • Triggers — a rule that says "when this event matches, send it there."
  • Targets — where the event is delivered.
Connection to functions: every event-driven Cloud Run function uses an Eventarc trigger under the hood. Eventarc is the plumbing that turns "a file landed in storage" into "my code runs."
"Routing" isn't one thing — it appears at every layer
LayerWhat "routing" means there
Pub/SubTopics fan messages out to subscriptions; messages route globally to the nearest permitted region.
Cloud RunTraffic routing — splitting requests across service revisions (e.g. 90/10 for a canary).
Airflow / ComposerTask routing — branch operators choose a path; executors route tasks to the right worker queue.
EventarcEvent routing — match events from many sources and deliver to the right handler.

So when someone says "routing" on this stack, ask which layer — message, request, task, or event.

🚦
A railway switchyard

Same trains, same tracks — but the points operator decides which track each train takes. Eventarc is the operator for events.

Mental model

One event format, many sources, delivered to the right handler. Routing in general: send each thing to its correct destination — whether that "thing" is a message, a request, a task or an event.

Onboarding gotchas
  • "Routing" is overloaded — always clarify the layer.
  • Eventarc rides on Pub/Sub, so Pub/Sub delivery semantics (at-least-once) apply to events too.
