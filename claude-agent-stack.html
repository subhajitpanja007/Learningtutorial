01 вҖ” Overview

What вҖңthe agent stackвҖқ actually is

Out of the box, Claude Code is a smart terminal that edits files and runs commands. The stack is the thin layer of config files and protocols that turns it into something that already knows your repo conventions, can reach your real systems, remembers across sessions, and behaves predictably вҖ” without you re-explaining everything every prompt.

Install Claude Code in 30 seconds

Everything below assumes Claude Code is installed. It needs Node.js 18+ and works in the terminal, VS Code, JetBrains, the desktop app, and on the web/iOS.

вҢҳ terminalbash
# install the CLI globally
npm install -g @anthropic-ai/claude-code

# run it inside any project folder
cd my-project

The seven layers, in the order they were built

You rarely need all of them. Most teams start with two (CLAUDE.md + one MCP server) and grow. Here is the whole timeline so you know what exists:

Nov 2024 В· MCP

Universal protocol that connects the agent to external tools & data вҖ” databases, GitHub, browsers, your docs.

Jul 2025 В· Subagents

Specialised assistants that run in their own isolated context window with their own tools & permissions.

Sep 2025 В· Hooks

Shell commands that fire automatically on lifecycle events вҖ” guardrails, formatters, logging, memory saves.

Oct 2025 В· Skills

Folders of reusable instructions the model loads on demand. The unified extensibility primitive.

Oct 2025 В· Plugins

Installable bundles that package skills + agents + hooks + MCP servers + commands into one unit.

Feb 2026 В· Agent Teams

Coordinated groups of subagents that collaborate on a shared task вҖ” the orchestration layer.

в—Ҷ The one rule that decides everything
If an instruction must be true on every single turn вҶ’ it belongs in CLAUDE.md. Everything else (workflows, tools, research, automation) lives in a more specialised layer so it doesnвҖҷt burn context when it isnвҖҷt needed.

03 вҖ” Native feature

CLAUDE.md project memory

Definition

A plain-Markdown file Claude Code loads into context automatically at the start of every session. It is the agentвҖҷs persistent memory of how this project works. Supports @path imports to pull in other files.

When / Why

Use it for facts that must be true on every turn: build & test commands, the package manager, code-style rules, architectural constraints, and your вҖңdefinition of doneвҖқ. Because it is always in context, keep it short and high-signal вҖ” it costs tokens on every message.

How to create

Run /init inside claude to auto-draft one from your codebase, then trim it hard. Or just create the file. Locations: ./CLAUDE.md (shared, committed), ./CLAUDE.local.md (personal, gitignored), ~/.claude/CLAUDE.md (global), and nested files per sub-directory.

CLAUDE.mdmarkdown
# Project: Acme Web App

## Commands
- Install: `pnpm install`
- Dev server: `pnpm dev`
- Test (run before saying done): `pnpm test`
- Lint + typecheck: `pnpm lint && pnpm typecheck`

## Conventions
- Package manager is **pnpm** вҖ” never npm/yarn.
- TypeScript strict mode; no `any`, no non-null `!`.
- React: function components + hooks only. Co-locate tests as `*.test.tsx`.
- Commits follow Conventional Commits.

## Constraints
- Never edit files in `src/generated/`.
- Never commit `.env*` or anything under `data/`.
- Always run the test suite before reporting a task complete.

## Context
@docs/architecture.md   # pull in extra detail only when needed
in-session memory commandsbash
/init        # generate a first-draft CLAUDE.md from the repo
/memory      # open CLAUDE.md in your editor to edit it
# Type a line starting with '#' during a chat to append a memory instantly:
# always prefer composition over inheritance

Three places it can live вҖ” machine, project, local

Every matching file is combined when a session starts, and the more specific file wins on conflicts. That gives you three layers (plus nested):

в‘  Machine / global
~/.claude/CLAUDE.md

Your personal defaults for every project on this computer вҖ” preferred languages, tone, global do/donвҖҷt rules. Never shared.

в‘Ў Project
./CLAUDE.md

Committed to the repo and shared with the whole team. The main one вҖ” commands, conventions and constraints for this codebase.

в‘ў Local
./CLAUDE.local.md

Project-specific but personal & git-ignored. Your own overrides, scratch notes or machine paths that shouldnвҖҷt reach the team.

в‘Ј Nested
./packages/api/CLAUDE.md

Drop one inside a sub-folder of a monorepo to add or override rules just for that part of the tree.

precedence вҖ” most specific winsbash
~/.claude/CLAUDE.md        # MACHINE  В· your personal defaults, every project
./CLAUDE.md                # PROJECT  В· committed, shared with the team
./CLAUDE.local.md          # LOCAL    В· personal & git-ignored, this repo only
./packages/api/CLAUDE.md   # NESTED   В· extra rules for one sub-folder
в—Ҷ When should you update CLAUDE.md again?
Update it the moment a fact changes or a pattern emerges вҖ” not on a fixed schedule. Concretely, edit it when you: add or rename a build / test / run command В· adopt a new convention, library or package manager В· catch Claude making the same mistake twice (encode the fix as one line, or just type # your rule mid-chat) В· add an MCP server, skill or subagent it should know about В· change the folder layout or your вҖңdefinition of doneвҖқ. Then prune anything now stale or obvious вҖ” a healthy CLAUDE.md trends shorter over time, not longer.
в–І keep it minimal вҖ” bloat actively hurts
Research in 2026 (Gloaguen et al.) found that over-stuffed, auto-generated context files reduce agent performance and inflate cost вҖ” the agent dutifully follows noise. Treat /init output as a draft. Drop generic вҖңarchitecture overviewвҖқ prose; keep commands, hard constraints, and non-obvious conventions.
Reference В· cheat sheet

Slash commands & the @ / # shortcuts

Inside the claude session, slash commands drive the tooling and two characters drive context: @ pulls a file in, # writes a memory out. Here are the ones youвҖҷll reach for daily.

Essential commands

/init

Scan the repository and generate a first-draft CLAUDE.md. Treat the output as a draft and trim it.

/memory

Open your CLAUDE.md (project or user) in an editor to edit it directly.

/compact

Summarise the conversation so far and continue with a smaller context window вҖ” exactly what frees up room in a long session. Add a hint, e.g. /compact focus on the open bug, to steer whatвҖҷs kept.

/clear

Wipe the current context and start a fresh conversation in the same project.

/help

List every available command in your build.

/model

Switch the model for this session (e.g. Opus вҶ” Sonnet).

Discover & manage the stack

/mcp

View connected MCP servers, their tools and auth status.

/agents

Create and manage subagents (name, description, tools, model).

/skills

List the skills currently available to the agent.

/plugin

Manage plugins & marketplaces вҖ” /plugin marketplace add owner/repo, then /plugin install name@repo.

Context: the @ and # characters

@path/to/file

Type @ at the prompt and an autocomplete appears вҖ” pick a file or folder to pull its contents into the current message. The fast way to say вҖңlook at thisвҖқ.

@AGENTS.md

The same @ used inside a CLAUDE.md is an import вҖ” that file is loaded with memory. This is how a one-line CLAUDE.md can defer to AGENTS.md.

# your rule

Start a chat line with # to append it to memory instantly вҖ” the quickest way to teach a convention without opening the file.

From the community tools

/skill-name

Invoke any installed skill on demand instead of waiting for the model to pick it up.

/caveman

(caveman) Toggle terse output to cut output tokens; /caveman lite|ultra, or say вҖңnormal modeвҖқ to stop.

/security-scan

(ECC) Run the AgentShield security review over your setup before shipping.

в—Ҷ The two-character mental model
@ reads, # writes. @ brings a file into this turnвҖҷs context; # sends a rule out to CLAUDE.md so it sticks on every future turn.
04 вҖ” Native feature

AGENTS.md the portable standard

Definition

An open, cross-tool standard (Linux-Foundation-stewarded, used by 60k+ repos) вҖ” a вҖңREADME for machinesвҖқ. The same file is read by OpenAI Codex, GitHub Copilot, Cursor, Google Gemini/Jules, Aider, Zed and Claude Code. It complements README.md; one targets humans, the other targets agents.

When / Why

Use it when you (or your team) hop between AI tools and donвҖҷt want to maintain five different config files. Put build/test commands, code style, conventions and constraints here once. In monorepos, drop a nested AGENTS.md in each package to override the root.

How to create

Create AGENTS.md at the repo root. The high-signal sections are: setup/build/test commands, code style, conventions, version pins, and programmatic checks. Skip long architecture essays вҖ” research shows they donвҖҷt improve behaviour.

AGENTS.mdmarkdown
# AGENTS.md

## Setup
- Node 20+. Install deps with `pnpm install`.

## Commands
- Build: `pnpm build`
- Test: `pnpm test`   # run before finishing any task
- Lint: `pnpm lint`

## Code style
- TypeScript strict. Named exports only.
- No comments unless explaining a non-obvious decision.

## Conventions
- Use the repository's existing patterns before introducing a new library.
- API routes live in `src/server/routes`; UI in `src/ui`.

## Checks an agent should run before declaring done
1. `pnpm typecheck`
2. `pnpm test`
CLAUDE.mdmarkdown
# Don't duplicate. Make AGENTS.md the source of truth
# and import it so Claude-specific notes sit alongside.

@AGENTS.md

## Claude-only notes
- Prefer the `code-review-graph` MCP tools before Grep/Read.
- When unsure, ask before deleting files.
в—Ҷ CLAUDE.md vs AGENTS.md
They coexist. AGENTS.md is the shared baseline every tool reads; CLAUDE.md is Claude-specific and takes precedence for Claude Code. The cleanest setup: put the real content in AGENTS.md, and make CLAUDE.md a one-liner that imports it via @AGENTS.md вҖ” single source of truth, zero drift.
05 вҖ” Native feature

instructions.md the convention layer

в–І honesty first
instructions.md is not an Anthropic-defined primitive like CLAUDE.md or Skills. ItвҖҷs a convention. Different ecosystems read differently-named instruction files. Treat вҖңinstructionsвҖқ as the umbrella for all of them, and avoid fragmenting your guidance across many files.
Definition

A human-readable rules/onboarding doc вҖ” sometimes literally instructions.md or CONTRIBUTING.md вҖ” plus the tool-specific instruction files each agent looks for: .github/copilot-instructions.md (Copilot), .cursor/rules/*.mdc (Cursor), .windsurfrules (Windsurf), and the standard AGENTS.md.

When / Why

Use a dedicated instructions file when you want one canonical вҖңhow we workвҖқ document that humans read and that your agent files import, or when a specific tool needs its own override file with rules that donвҖҷt belong in the shared standard.

How to create

Write the doc, then reference it from your agent config via @import (Claude) or let the tool auto-discover its native path. Pick AGENTS.md as the source of truth and point the others at it.

.github/copilot-instructions.mdmarkdown
# Copilot instructions
- This is a pnpm monorepo. Build with `pnpm -r build`.
- Use the patterns already present in the package you are editing.
- All new code is TypeScript strict; add Vitest tests for new logic.
.cursor/rules/conventions.mdcmarkdown
---
description: Project-wide conventions
alwaysApply: true
---
- Constructor injection only; never field injection.
- Keep functions under ~40 lines; extract helpers.
- Mirror the existing folder structure when adding files.
CLAUDE.mdmarkdown
# One source of truth, imported everywhere
@AGENTS.md
@docs/instructions.md   # the human-facing "how we work" doc
в—Ҷ rule of thumb
One canonical doc, many thin pointers. Duplicated instructions across files drift out of sync and confuse agents вҖ” let AGENTS.md hold the truth and have tool-specific files import or restate only whatвҖҷs genuinely tool-specific.
06 вҖ” Native feature

Agent Skills SKILL.md

Definition

A folder containing a SKILL.md (YAML frontmatter with a name + description, then Markdown instructions) and optional scripts/resources. ItвҖҷs an open standard (Anthropic, Oct 2025; open-sourced and now adopted by Codex, Cursor, VS Code and 30+ tools). Claude loads it on demand when the description matches the task, or you invoke it with /skill-name.

When / Why

For repeatable workflows and methodology: a code-review checklist, a release dance, a migration generator, your data-analysis routine. Skills are cheap вҖ” about 100 tokens of context until invoked вҖ” so you can install fifty and pay almost nothing for the ones you arenвҖҷt using (progressive disclosure).

How to create

Make a folder under .claude/skills/<name>/ (project) or ~/.claude/skills/ (personal) and add SKILL.md. Install community skills with the skills CLI: npx skills add <owner/repo>.

.claude/skills/tree
.claude/
в””в”Җв”Җ skills/
    в””в”Җв”Җ review-component/
        в”ңв”Җв”Җ SKILL.md          # frontmatter + instructions (always read)
        в”ңв”Җв”Җ checklist.md      # extra detail, loaded only if referenced
        в””в”Җв”Җ scripts/
            в””в”Җв”Җ run-axe.sh     # optional executable helpers
SKILL.mdmarkdown
---
name: review-component
description: >
  Review a React component for accessibility, re-render
  performance, and test coverage. Use when the user asks to
  review, audit, or harden a .tsx component.
---

# Component review

When invoked, do the following in order:

1. Read the component and its test file.
2. Check accessibility: roles, labels, keyboard nav, focus order.
3. Flag re-render risks: inline object/array props, missing
   `useMemo`/`useCallback`, unstable keys.
4. Confirm a Vitest test exists for each branch; if not, write one.
5. Output a numbered findings list, most severe first.
вҢҳ terminalbash

mkdir -p .claude/skills/review-component
$EDITOR .claude/skills/review-component/SKILL.md

# install from the community registry (skills.sh)
npx skills add anthropics/skills

# inside claude: list skills, then invoke one
/skills
/review-component src/ui/Button.tsx
в—Ҷ Skill vs MCP vs Subagent
Skill = shapes behaviour/knowledge inside the tools the agent already has. MCP = adds a brand-new external capability (a database, an API). Subagent = an isolated worker with its own context. A skill can teach Claude how to review auth code; it canвҖҷt, by itself, connect to your database вҖ” thatвҖҷs MCPвҖҷs job.
07 вҖ” Native feature

MCP Servers the USB-C for AI tools

Definition

The Model Context Protocol is an open standard that lets agents call external tools and data sources through one universal interface вҖ” databases, GitHub, Slack, the filesystem, browsers, monitoring stacks. A server exposes a set of tools; Claude calls them like any other tool.

When / Why

When the agent needs live access to a system instead of you pasting data into the chat вҖ” querying your MongoDB, opening a PR, reading your issue tracker. It eliminates the вҖңcopy context in by handвҖқ loop.

How to create

Add servers via the CLI, or commit a .mcp.json file so the whole team shares the same setup. Scopes: local (just you), project (committed), user (all your projects). Manage live with /mcp.

.mcp.jsonjson
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "./"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}" }
    },
    "MongoDB": {
      "command": "npx",
      "args": ["-y", "mongodb-mcp-server@latest", "--readOnly"],
      "env": { "MDB_MCP_CONNECTION_STRING": "mongodb://localhost:27017/myDatabase" }
    }
  }
}
вҢҳ terminalbash
# add a stdio server (local process)
claude mcp add github -- npx -y @modelcontextprotocol/server-github

# add a remote HTTP server
claude mcp add --transport http notion https://mcp.notion.com/mcp

# add MongoDB at project scope so the team inherits it
claude mcp add MongoDB --scope project -- \
  npx -y mongodb-mcp-server@latest --readOnly

# list / inspect connected servers (also works as /mcp inside claude)
claude mcp list
в–І MCP is a trust + cost surface
Only connect servers you trust, and prefer least privilege вҖ” start --readOnly, scope tokens tightly, never hand a staging-DB server blanket write access вҖңfor convenienceвҖқ. Every connected server also adds tool-schema tokens to every session, so prune ones you donвҖҷt use (the codeburn tool below finds the dead weight).
08 вҖ” Native feature

Subagents В· Hooks В· Plugins the automation layer

Subagents

Specialised assistants that run in their own context window with their own prompt, tool access and permissions вҖ” perfect for noisy research, large-file reading, or parallel code review you donвҖҷt want polluting your main context. Create them with /agents or a Markdown file. Agent Teams (2026) coordinate several at once.

.claude/agents/security-reviewer.mdmarkdown
---
name: security-reviewer
description: Audits diffs for injection, authn/z and secret-leak issues.
tools: Read, Grep, Bash
model: opus
---
You are a security reviewer. Inspect only the changed files.
Report findings as: severity, file:line, the risk, and a fix.
Do not modify code; produce a review only.

Hooks

Shell commands bound to lifecycle events вҖ” PreToolUse (validation gates), PostToolUse (lint/format/tests after an edit), SessionStart, PreCompact, Stop. They run automatically, no prompting. Configure in .claude/settings.json.

.claude/settings.jsonjson
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "pnpm prettier --write \"$CLAUDE_FILE_PATHS\"" }
        ]
      }
    ]
  }
}

Plugins

Installable bundles that package skills + subagents + hooks + MCP servers + slash-commands into one unit, distributed through a marketplace. Convenient вҖ” and a bigger trust surface, since youвҖҷre installing hook logic and tool connections, not just prompts. Vet before installing.

inside claudebash
# add a marketplace, then install a plugin from it
/plugin marketplace add owner/repo
/plugin install some-plugin@repo

# browse what's installed
/plugin

# Acme API (Spring Boot 3 / Java 21)
## Commands
- Build+test (must pass): `./mvnw test`
- Run: `./mvnw spring-boot:run`
## Rules
- Always use the Maven wrapper `./mvnw`, never a global mvn.
- Layered flow only: controller -> service -> repository.
- **Constructor injection** only; never `@Autowired` on fields.
- Validate request bodies with Jakarta `@Valid`.
- One `@SpringBootTest` (or slice test) per new endpoint.
- Keep controllers thin; business rules live in services.

MongoDB В· the data layer

MongoDB is the database behind the stacks above. The leverage here is the official MongoDB MCP server вҖ” it lets Claude inspect collections, reason about your real schema, and write safe queries, instead of guessing from code.

Definition

MongoDB ships an official MCP server (mongodb-mcp-server, a Node package, also a Docker image). It connects to a local instance or an Atlas cluster and exposes tools to list collections, read schemas, run queries and aggregations.

When / Why

Whenever Claude works on data-backed features. With the server connected it can confirm field names, propose indexes, and draft aggregation pipelines against the actual shape of your data вҖ” eliminating a whole class of guesswork bugs.

How to create

Add it to .mcp.json with a connection string, or install the official MongoDB plugin bundle (Claude Code / Codex / Cursor / Gemini) which ships the server and pre-built MongoDB agent skills. Start read-only.

.mcp.json вҖ” local, read-onlyjson
{
  "mcpServers": {
    "MongoDB": {
      "command": "npx",
      "args": ["-y", "mongodb-mcp-server@latest", "--readOnly"],
      "env": { "MDB_MCP_CONNECTION_STRING": "mongodb://localhost:27017/acme" }
    }
  }
}

CLAUDE.md вҖ” database guardrailsmarkdown
# Database rules
- The MongoDB MCP server is **read-only** by default вҖ” keep it so
  unless a task explicitly needs writes.
- Never run destructive operations (drop/deleteMany) without
  showing the query and asking for confirmation first.
- Propose an index before suggesting any query that would collection-scan.
- Schema changes go through a migration script, never ad-hoc edits.
