What you query is thetip of the iceberg.

Apache Iceberg is an open table format — a metadata spec that turns a pile of Parquet files in object storage into a real table with ACID transactions, schema evolution, hidden partitioning, and time travel. The engine sees a tidy table on the surface. Iceberg manages the vast structured mass below.

The mental model

Three things to never confuse

Most Iceberg confusion comes from mixing these layers. Click each to open the deep dive.

Layer 1 · bytes

File format

How a single file's bytes are laid out. Parquet, ORC, Avro. Columnar, compressed, splittable. Iceberg uses these — it does not replace them.

Layer 2 · the table

Table format

How many files add up to one table over time: schema, snapshots, partitions, stats, ACID. This is Iceberg. Also: Delta Lake, Hudi.

Layer 3 · the pointer

Catalog

The thing that maps a table name to its current metadata pointer, atomically. REST, Hive, Glue, Nessie, Polaris — on GCP, the Lakehouse runtime catalog.

Why it exists: object stores (GCS, S3) have no atomic multi-file commit and no "rename folder." Hive tables tracked partitions as directories, which made schema/partition changes a full rewrite and made concurrent writes unsafe. Iceberg moves the source of truth into immutable metadata files + one atomic pointer swap, so a commit is all-or-nothing.

The layered model

Data Lake vs Data Lakehouse — layer by layer

A lake is just storage + compute. A lakehouse inserts two layers between them — a table format and a catalog — and those two layers are the entire difference. They're what turn a pile of Parquet files into a real, transactional table.

The two purple layers are everything. The table format (Iceberg) records which files make up the table right now, their stats, and lets you change it atomically — giving ACID, time travel, and safe concurrent writes. The catalog is the front desk that points every engine to the table's current metadata. Same cheap storage, same engines — but now it behaves like a warehouse.
Three eras

Warehouse, Lake, Lakehouse — and where Iceberg sits

Forty years of analytics in three moves. Each architecture solved the previous one's pain and created a new one. Iceberg is the reliability layer that lets the newest one exist. Click any card for the deep dive, the analogy, and the mental model.

The two-tier problem Iceberg kills: because lakes weren't reliable, teams ETL'd a curated copy into a warehouse — two systems, double storage, constant pipelines, stale data. The organizing pattern

Medallion — and why Iceberg lives in every layer

Bronze → Silver → Gold is a quality ladder: raw fidelity becomes trusted business value one hop at a time. Iceberg is the table format at all three tiers — not just one — because ACID, schema evolution, and time travel matter at each.

Raw

Bronze

Ingest source data as-is + load metadata. Append-only, schema-drift tolerant, partitioned by ingest time. The immutable canonical record.

Validated

Silver

Cleansed, deduplicated, conformed, typed. The enterprise view of key entities. PII masked here so Silver+ is safe-by-default.

Curated

Gold

Aggregates, marts, ML features — tuned per consumer. On GCP, often BigQuery-native for performance and governance.

Why Iceberg at every tier: without ACID, Silver/Gold silently drift from Bronze. With Iceberg you get atomic loads at each layer, time travel to debug a bad transform against the exact prior Bronze snapshot, schema changes that flow downstream without breaking pipelines, and instant rollback of a bad batch.

Layering

Iceberg on top of Parquet

Parquet = data file. Iceberg = the table/metadata layer above it.

Iceberg ≠ Parquet. Parquet is the file format (the bytes). Iceberg is the table format (the metadata that says which Parquet files form the table, their stats, and the current snapshot). Iceberg prunes whole files via its manifests before Parquet even opens a footer; Parquet then skips row groups inside the survivors.
History · what came before

Apache Hive — the original SQL-on-Hadoop

Hive (≈2008) let analysts query Hadoop with SQL instead of MapReduce. "Hive" means three different things — and its table format is exactly what Iceberg was built to replace.

1 · Engine

Hive (warehouse)

The SQL layer + execution engine (HiveQL on MapReduce/Tez).

2 · Table format

Hive tables

Files in directories; partition values encoded in folder paths like /city=Mumbai/; partitions found by listing directories.

A database of table/partition metadata — i.e. a catalog. Still widely used in 2026.

The other open formats

Apache Hudi — the upsert & streaming format

Hudi ("Hadoop Upserts Deletes and Incrementals", from Uber, 2016) is the third open table format. Where Iceberg shines at multi-engine analytics and Delta at Databricks, Hudi is built for constant updates — CDC, streaming, fast upserts/deletes.

Copy-on-Write (CoW)

Rewrite on update

Changed rows rewrite their Parquet file. Read-optimized, higher write cost. Like Iceberg/Delta's default.

Merge-on-Read (MoR)

Append a change log

Updates go to row-based log files, merged with base Parquet at read time (or by async compaction). Lowest write latency.

How it finds the rowHudi keeps a timeline of commits and a record-level index mapping each key → the file it lives in, so a single changed customer can be updated without scanning the whole table. That index is what makes Hudi fast for upsert-heavy CDC.

03 · Architecture

The official three-layer architecture

Straight from the Apache Iceberg spec: a table is a catalog pointing at a metadata tree that points at data files. Everything except the catalog pointer is an immutable file in object storage. Click a layer to light it up.

Layer 1 · Catalog

The pointer / source of truth

Maps each table name → the location of its current metadata.json. A commit is an atomic compare-and-swap of this one pointer. On GCP: the Lakehouse runtime catalog (Iceberg REST).

db.customers → metadata-00007.json
▼ points to current metadata
Layer 2 · Metadata

The immutable metadata tree

metadata.json (schema w/ column IDs, partition specs, all snapshots) → manifest list (one per snapshot, with partition ranges) → manifest files (list data files + column min/max/null stats). This is what makes pruning and time travel possible.

▼ lists
Layer 3 · Data

The data files

The actual rows, in Parquet (or ORC/Avro), in GCS/S3/HDFS. Iceberg never rewrites these on metadata changes. Format V2/V3 add small delete files / deletion vectors for row-level changes.

Tap Catalog, Metadata, or Data above to see what each layer stores and why it matters.
Mental model: the metadata tree is a book's table of contents → chapter summaries → pages. To find "errors on 2026-06-25" you never read every page; the contents (manifest list) jumps you to the right chapter (manifest), whose summary (column stats) jumps you to the right page (Parquet file). A date filter can skip ~99% of files before any data is read — warehouse-class speed on plain object storage.

02 · Schema evolution

Change the schema. Touch zero data.

Iceberg tracks every column by a unique, never-reused field ID, not by name or position. Names and order live only in metadata, so renaming or reordering is a metadata edit. New columns read back as NULL for old files — or a default value in V3.

On bronze.customers — every op is metadata-only
-- add a column — old rows read NULL (or a V3 default), no rewrite ALTER TABLE bronze.customers ADD COLUMN loyalty_tier STRING; -- rename — field ID unchanged, existing Parquet never opened ALTER TABLE bronze.customers RENAME COLUMN full_name TO customer_name; -- widen a type (safe promotion only) ALTER TABLE bronze.customers ALTER COLUMN customer_id TYPE BIGINT; -- int → long -- reorder for readability (display order is metadata) ALTER TABLE bronze.customers ALTER COLUMN email AFTER customer_name; -- V3: a default applied to OLD and NEW rows with no rewrite ALTER TABLE bronze.customers SET TBLPROPERTIES ('format-version'='3'); ALTER TABLE bronze.customers ADD COLUMN region STRING DEFAULT 'APAC'; -- drop — the field ID is retired FOREVER and never reused ALTER TABLE bronze.customers DROP COLUMN loyalty_tier;

The field-ID guarantee in action: if you DROP COLUMN pan then later ADD COLUMN pan, the new pan gets a fresh ID — old files store the old ID, so the re-added column reads NULL, never the stale PANs. Nested struct/map/list fields evolve by the same rules. (On GCP's Lakehouse runtime catalog, table RENAME itself isn't supported; txn_amount is a DOUBLE, so moving it to decimal needs a new column + backfill, not a promotion.)
The trick

Names are a lie the metadata tells

Inside data files, columns are stored by ID. The mapping id → name sits in metadata.json. So RENAME email TO email_address rewrites one JSON line. Old Parquet files are never opened.

Why this matters: on Hive, adding a column in the middle could silently shift every downstream column by position. Iceberg makes that class of bug impossible.
V3 upgrade · 1.10+

Default values, finally

Adding a non-nullable column used to force a rewrite. In format V3 you set initial-default (for existing rows) and write-default (for new rows) — both as a metadata change.

ALTER TABLE sales.orders ADD COLUMN channel STRING DEFAULT 'web'; -- V3, no rewrite

03 · Partitioning

Partitions you can't forget — and can change later

Iceberg's two partitioning superpowers: hidden partitioning (the layout is derived from a column by a transform, so queries never need a special partition column or filter) and partition evolution (change the scheme without rewriting history).

On bronze.customers
-- partition by day of ingestion + 16 hash buckets of city CREATE TABLE bronze.customers (...) USING iceberg PARTITIONED BY (days(ingested_at), bucket(16, city)); -- query filters the REAL column; Iceberg prunes automatically SELECT city, SUM(txn_amount) FROM bronze.customers WHERE ingested_at >= '2026-06-02' GROUP BY city; -- partition EVOLUTION (metadata-only): monthly → daily, no rewrite ALTER TABLE bronze.customers DROP PARTITION FIELD months(ingested_at); ALTER TABLE bronze.customers ADD PARTITION FIELD days(ingested_at);

Inspect the layout via metadata tables

-- per-partition file counts (spot skew & over-partitioning) SELECT partition, record_count, file_count, spec_id FROM bronze.customers.partitions ORDER BY file_count DESC; -- per-file stats that drive data skipping SELECT file_path, record_count, file_size_in_bytes, lower_bounds, upper_bounds FROM bronze.customers.files;
The over-partitioning trap: identity(customer_id) would make a partition per customer → millions of tiny files and bloated metadata. Use bucket(N, customer_id) for high-cardinality keys and aim for 128 MB – 1 GB per file. Because partition evolution is metadata-only, old files keep their old spec and queries plan correctly across both.
Superpower 1

Hidden partitioning

You declare PARTITIONED BY day(event_ts). Iceberg stores the transform in the spec and applies it. A query that filters event_ts > '2026-06-01' is pruned automatically — no event_date column, no human discipline required.

Superpower 2

Partition evolution

Traffic grew, so you switch from month() to day(). Iceberg records a new partition spec going forward; old data keeps its old layout. Both are pruned correctly in one query.

Transform playground

The built-in transforms. V3 also allows multi-argument transforms.

identityyear()month() day()hour()bucket(N, col) truncate(W, col)void()
High-cardinality keys

bucket(N)

Hashes a column into N buckets — perfect for user_id or pan where date partitioning makes no sense. Spreads load, enables bucketed joins.

Strings & numbers

truncate(W)

Keeps the first W chars/units — e.g. truncate(4, zipcode). Cheaper grouping than full identity.

The classic mistake

Over-partitioning

Partitioning a small table by hour() creates millions of tiny files. Aim for partitions of ~100MB–1GB, not "one folder per row."

silver_merge.py · CDC upsert + PAN masking 
# Silver = cleansed + governed. Mask PAN here so Silver/Gold are safe-by-default.
from pyspark.sql.functions import col, sha2, concat, lit, expr

bronze = spark.table("bronze.customers")

masked = (bronze
  # partial mask for display:  ABCPK1234K -> ABXXXXXX4K
  .withColumn("pan_masked",
       concat(expr("substring(pan,1,2)"), lit("XXXXXX"),
              expr("substring(pan,9,2)")))
  # deterministic token for safe joins (salt from Secret Manager)
  .withColumn("pan_token", sha2(concat(col("pan"), lit(SALT)), 256))
  .drop("pan"))          # raw PAN never written to Silver

# Idempotent CDC upsert — V3 writes a deletion vector, not a file rewrite
masked.createOrReplaceTempView("updates")
spark.sql("""
MERGE INTO silver.customers t
USING updates s ON t.customer_id = s.customer_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *""")

    The pruning ladder

    How a query gets fast — what gets skipped, in order

    1. Partition pruning — the manifest list skips whole manifests whose partition range can't match (e.g. wrong day).
    2. File skipping — each manifest skips data files whose column min/max can't match the filter.
    3. Bloom filters — for high-cardinality equality like pan = '…', skip row groups the value definitely isn't in.
    4. Predicate + column pushdown — read only matching row groups, and only the columns you selected.
    5. Read the survivors — vectorized Parquet scan of the few files left. A good date filter skips ~99% before any data I/O.

    Rule 1

    Never log it

    No .show(), no print, no error message that echoes raw PAN. Logs leak to Cloud Logging.

    Rule 2

    Mask at the boundary

    Transform PAN as it crosses Bronze→Silver. Raw column stops at Bronze; Silver carries only pan_token/pan_masked.

    Rule 3

    Salt from a vault

    Hash salts live in Secret Manager, never in code or notebooks. Rotate on a schedule; a leaked unsalted hash is reversible by brute force.

    What changed, and the best approach now

    Format V3 went GA in Iceberg 1.10 (Sep 2025) and matured through 1.11 (May 2026). It's the biggest spec jump since V2. Here's what's new and what to actually do.

    Headline · DML

    Deletion vectors

    Roaring-bitmap Puffin files replace V2 position-delete files. O(1) read-time lookup, up to ~10× faster updates/deletes. The reason V3 exists.

    CDC built in

    Row lineage

    _row_id + _last_updated_sequence_number per row — native change tracking without external CDC tooling.

    Semi-structured

    VARIANT type

    Binary semi-structured column with shredding + predicate pushdown. Retires the "JSON-as-STRING" workaround.

    New types

    Geo + nanosecond ts

    Native geometry/geography and nanosecond-precision timestamps. Plus multi-argument partition transforms.

    Schema

    Default values

    Add non-nullable columns with initial-default / write-default — no rewrite. Covered in §02.

    Read carefully

    Compatibility

    V3 readers read V2; V2 readers cannot read V3. Upgrade is one-way. Check every engine in your stack first.

    Engine reality check (early–mid 2026): Spark 4.0 is the V3 reference implementation (Spark 3.5.x partial via Iceberg 1.9.1+); Flink 2.0 + Iceberg 1.10 is strong for streaming CDC. On GCP, BigQuery's V3 support (incl. deletion vectors) was in preview. Trino V3 support lagged. If any engine in your pipeline can't read V3, stay on V2 for that table until it can — a table is only as portable as its least-capable reader.

    Production checklist

    Always on

    Hygiene jobs

    Schedule rewrite_data_files, rewrite_manifests, expire_snapshots and remove_orphan_files. Target 128MB–1GB files. Unmanaged Iceberg degrades into a small-files swamp.

    Catalog

    Go REST

    Prefer the Iceberg REST catalog (Lakehouse runtime catalog on GCP) over Hadoop/Hive catalogs. Credential vending + one source of truth across engines.

    Writes

    MoR for churn

    Set merge-on-read for high-update tables to exploit deletion vectors; keep copy-on-write for read-heavy, rarely-updated tables.

    Governance

    Tag at birth

    Attach policy tags to PII columns in Bronze and let column-level security + masking flow downstream. Govern at ingestion, not as an afterthought.
