The cast

Three managed services, one job

Each does exactly one thing. The art is wiring them together so a schedule reliably turns into processed data — with retries, identity, and decoupling baked in.

When · the trigger

Cloud Scheduler

A fully-managed cron service. On a schedule you define, it fires one of three target types: publish to a Pub/Sub topic, call an HTTP endpoint, or hit App Engine. It only triggers — it never processes data itself.

Fires bycron expression + IANA timezone
TargetsPub/Sub · HTTP · App Engine
Reliabilityconfigurable retry & backoff
What · the messenger

Pub/Sub topic

An asynchronous message bus. A publisher drops a message on a topic; one or more subscriptions deliver it to consumers. It decouples the trigger from the worker, absorbs bursts, and guarantees at-least-once delivery.

Topicwhere messages are published
Subscriptionpull · push · BigQuery · GCS
Safety netdead-letter + retry policy
How · the worker

Cloud Run Job

A container that runs a task to completion and exits — not a server. Perfect for batch ingest. It runs your PySpark code, reads the on-prem source, and writes Parquet. Splits into parallel tasks when you need them.

Unitexecution → one or many tasks
Identityruns as a service account
ReachVPC egress to on-prem

Frame by frame

One full run, step by step

This is the default path (scenario A) for the SOR pipeline, broken into the exact sequence of events from clock tick to “done”.

The clock fires

At 0 2 * * * (02:00 Asia/Kolkata) Cloud Scheduler triggers. It publishes a small JSON message — {"source":"sor","run_date":"2026-06-16"} — to the topic ingest-trigger.

Pub/Sub accepts & fans out

The topic stores the message and immediately offers it to every attached subscription. The push subscription (Eventarc) targets the Cloud Run Job's run endpoint, carrying an OIDC token for authentication.

The job execution starts

Cloud Run creates an execution with one or more tasks. Each task is your container, booting with env vars CLOUD_RUN_TASK_INDEX and CLOUD_RUN_TASK_COUNT so it knows its slice of work.

PySpark reaches on-prem

Through Serverless VPC egress over Cloud VPN / Interconnect, PySpark connects to the on-prem SOR system (JDBC or file mount), reads the raw rows, and builds a DataFrame. Secrets come from Secret Manager — never hard-coded.

Parquet lands in the raw zone

PySpark writes columnar Parquet, partitioned by date, to gs://sor-raw-zone/... with snappy compression. Schema travels inside the files, so downstream readers need no DDL.

Load & announce

A BigQuery load job (or external table) exposes the Parquet for SQL. The job then publishes ingest.complete back to a topic — the “done” event that lets the next stage start. The container exits 0; the task is marked succeeded.

target Choosing what Scheduler fires

  • Pub/Sub target — topic name + a message body + optional attributes. Best for decoupling (scenarios A, C, D, E).
  • HTTP target — URL, method, headers, body, and an OIDC token (for Cloud Run / Functions) or OAuth token (for *.googleapis.com). This is how you call a Cloud Run Job's :run endpoint directly (scenario B).
  • App Engine — legacy in-app handlers.

retry When a fire fails

  • retryCount — max attempts before giving up.
  • maxRetryDuration — stop retrying after this window.
  • minBackoffDuration / maxBackoffDuration — wait range between attempts.
  • maxDoublings — how fast the backoff grows.
  • attemptDeadline — how long one attempt may run.

push Pub/Sub delivers to you

  • Pub/Sub POSTs the message to your HTTPS endpoint (Cloud Run / Function).
  • No infrastructure to run while idle — fully event-driven; this is what Eventarc uses.
  • You return 2xx to acknowledge; anything else triggers redelivery with backoff.

pull You fetch from Pub/Sub

  • A long-running worker (Cloud Run service / GKE) asks for messages and acks them.
  • Best for high throughput, custom flow-control, and ordered processing.
  • Ack deadline up to 600 s; unacked messages are redelivered.

safety Nothing gets silently lost

  • Dead-letter topic — after maxDeliveryAttempts, poison messages move to a DLQ for inspection.
  • Retry policy — immediate or exponential backoff between redeliveries.
  • Exactly-once delivery — opt-in to suppress duplicates within the ack window.

order & filter Control what arrives

  • Ordering key — messages with the same key are delivered in publish order.
  • Filters — a subscription can receive only messages matching an attribute, e.g. attributes.source = "sor".
  • Retention — replay messages for up to 7 days (configurable to 31).

from pyspark.sql import SparkSession
import os, datetime

spark = SparkSession.builder.appName("sor-ingest").getOrCreate()
run_date = datetime.date.today().isoformat()

# 1 · read raw SOR data from on-prem (JDBC over private VPC egress)
df = (spark.read.format("jdbc")
        .option("url",    os.environ["SOR_JDBC_URL"])
        .option("dbtable", "erp.orders")
        .option("user",   os.environ["SOR_USER"])
        .option("password", os.environ["SOR_PWD"])  # from Secret Manager
        .load())

# 2 · land RAW as Parquet, partitioned by date, snappy compressed
(df.write
   .mode("overwrite")
   .partitionBy("dt")
   .option("compression", "snappy")
   .parquet(f"gs://sor-raw-zone/source=erp/table=orders/dt={run_date}"))

# 3 · announce completion → next stage listens on this topic
publish("ingest.complete", {"source": "sor", "dt": run_date})
spark.stop()  # container exits 0 → task succeeded

Where Spark actually runs. For modest volumes, PySpark runs inside the job's container (local Spark). For truly large distributed loads, the job instead submits a Dataproc Serverless batch and waits — same trigger, heavier engine. Either way the output contract is identical: Parquet in the raw zone.

tasks Execution → tasks → parallelism

  • One execution launches --tasks N container instances.
  • --parallelism caps how many run at once.
  • Each reads CLOUD_RUN_TASK_INDEX / …_COUNT to claim its data slice.
  • Per-task --task-timeout and --max-retries handle stragglers.

reach & identity Talking to on-prem safely

  • Image pulled from Artifact Registry; CPU/memory set per job.
  • Direct VPC egress or a Serverless VPC connector routes to private IPs over Cloud VPN / Interconnect.
  • Runs as a dedicated service account; credentials from Secret Manager.

The fine print that makes it production-grade

Tiny details worth knowing

The small configuration choices that separate a demo from a pipeline you can sleep through the night with.

IAM Who is allowed to do what

  • Scheduler's service account needs roles/pubsub.publisher (Pub/Sub target) or roles/run.invoker (direct HTTP).
  • The Cloud Run Job's service account needs read on the source, roles/storage.objectAdmin on the bucket, and roles/bigquery.dataEditor.
  • Push subscriptions use an OIDC token tied to a dedicated invoker account.

idempotency Surviving redelivery

  • Pub/Sub is at-least-once — the same message can arrive twice.
  • Write to a dated partition with mode("overwrite") so a re-run is safe.
  • Carry a run_date / run-id so duplicate executions are detectable.

env vars What the container knows

  • CLOUD_RUN_TASK_INDEX — this task's number (0-based).
  • CLOUD_RUN_TASK_COUNT — total tasks in the execution.
  • CLOUD_RUN_EXECUTION · CLOUD_RUN_TASK_ATTEMPT — for logging & retries.

observability Knowing it actually ran

  • Cloud Logging captures every task's stdout/stderr; filter by execution id.
  • Cloud Monitoring alerts on failed executions or unacked-message age.
  • The ingest.complete event doubles as an audit trail and a stage gate.

orchestration Chaining many stages

  • Choreography — each job publishes a “done” event the next subscribes to (loose, scalable).
  • Cloud Workflows — central control for ordered, conditional, multi-step DAGs.
  • Mix both: Scheduler kicks off, events fan work out, Workflows guards the critical path.

cost & scale Paying only for the run

  • Cloud Run Jobs bill per task vCPU-second — zero cost while idle.
  • Pub/Sub bills per message volume; Scheduler is a few cents per job/month.
  • Parquet + partition pruning keeps BigQuery scan costs down.
Recap

Clock → message → container → Parquet → next stage

Cloud Scheduler decides when. Pub/Sub carries the what and decouples everyone. The Cloud Run Job is the how — it wakes up, pulls SOR & Platform data, lands Parquet with PySpark, and rings the bell for whatever comes next. Five wiring patterns, one dependable shape.
