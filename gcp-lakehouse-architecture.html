STEP 1 / 10

Medallion architecture — Bronze, Silver, Gold

Instead of cleaning data in one giant risky step, a lakehouse refines it through three zones. Each zone has one job and one quality promise. If anything breaks downstream, you can always replay from Bronze — the untouched original.

🥕 Cooking analogy: Bronze = groceries straight from the farm (dirt and all), Silver = washed, peeled and chopped ingredients, Gold = the finished plated dish served to guests.
data flow records getting cleaner ⚙ Spark transformation

▶ Click a zone or a question chip for the full beginner explanation.

Design rule 01

Never edit Bronze

Bronze is your insurance policy. Because it's an untouched, append-only copy of the source, you can delete and rebuild Silver and Gold any time a bug slips through — without re-asking the source systems for data.

Design rule 02

Silver is the contract

Teams agree that anything in Silver is deduplicated, correctly typed and validated. Analysts and ML engineers build on Silver without re-checking the basics — that shared trust is what makes the platform fast.

Design rule 03

Gold speaks business

Gold tables are named after business concepts (daily revenue, customer churn), not source systems. They're small, pre-aggregated and fast — a dashboard should never need to scan billions of raw rows.

BigLake — one secure table layer over your data lake

Your files live once in Google Cloud Storage. BigLake wraps them in a governed "table layer" so BigQuery, Spark and open engines all see the same tables with the same security — no copies, no exports, no per-engine permission chaos. Watch the queries beam down and rows beam back up.

📚 Library analogy: GCS is the book warehouse, BigLake is the librarian's front desk. Nobody wanders the shelves directly — every reader (BigQuery, Spark, Trino) asks the desk, which checks their library card (row/column security) and fetches pages fast using its catalogue (metadata cache).

The problem

Before BigLake: two worlds

Warehouse data was secure but expensive to copy in; lake files were cheap but had only coarse bucket-level permissions. Teams duplicated data between both — twice the storage, twice the pipelines, and security drift everywhere.

How a query flows

Four quick hops

1) You run SQL in BigQuery. 2) BigLake checks your row/column permissions. 3) The connection's service account reads only the needed Parquet bytes from GCS (helped by the metadata cache). 4) Filtered rows stream back. You never get raw file access.

When to use

Lake data, warehouse manners

Use BigLake whenever data should stay in open formats on GCS (for Spark, cost, or multi-engine reasons) but must still be queryable in BigQuery with real governance — exactly our Silver and Gold Iceberg tables.

Apache Iceberg — database superpowers for plain files

Parquet files alone can't be safely updated, can't evolve their schema, and don't remember their history. Iceberg is an open table format: a clever stack of metadata files that sits on top of your Parquet and gives it ACID transactions, time travel and painless schema changes — while staying open to every engine.

🚢 Iceberg analogy: what engines "see" above the waterline is a simple table. Below the waterline sits the metadata machinery — snapshots, manifests, statistics — silently doing the heavy lifting. Data files are never edited; every change just publishes a new snapshot pointing at a new list of files.

Mental model 01

Plans, not loops

Transformations only describe a recipe. The first action (write, count, MERGE) hands the whole recipe to Catalyst, which optimizes it before a single byte is read.

Mental model 02

Stages end at shuffles

Anything per-row is cheap and stays in one stage. Anything that needs "all rows with the same key together" forces a shuffle — the slowest, most expensive moment of the job.

Mental model 03

Parallelism = partitions

You can never use more cores than you have partitions. Too few → idle executors; thousands of tiny ones → scheduling overhead. Aim ~100–200 MB per partition and let AQE tidy up.

Rule of thumb 01

Pin the connector version

Image 2.1+ ships a built-in connector that overrides plain --jars. Set it explicitly: --metadata SPARK_BQ_CONNECTOR_VERSION=0.44.2 on clusters, or --properties dataproc.sparkBqConnector.version=0.44.2 on Serverless.

Rule of thumb 02

Default to indirect writes

For large batch loads and partitioned/clustered targets, choose indirect + Avro staging. Reserve direct (Storage Write API) for smaller, latency-sensitive writes — and only on connector ≥ 0.24.2.

Rule of thumb 03

Read Arrow, push column & predicate filters

Arrow is the connector's default and the lighter wire format. Combine with .option("filter", "_PARTITIONDATE = …") to prune partitions — never .where() on a pseudo-column.

Money saver 01

Make the filter mandatory

Set require_partition_filter = TRUE on big tables. BigQuery refuses unfiltered queries with a clear error — turning a $200 oops into a 1-line fix.

Money saver 02

Cache & materialized views

Identical queries served from cache for 24h — $0, milliseconds. Hot aggregates promoted to MVs are auto-rewritten into matching queries, no app changes needed.

Money saver 03

Avoid SELECT * & CAST gymnastics

Star-select forces every column off disk. Casting partition columns or filtering pseudo-columns the wrong way silently disables pruning. Name your columns, prune your dates, and dry-run first.

▶ Click either column — or the shared storage — for the deep dive on how each engine plans, parallelises and reads.

BigQuery side

You write SQL; Google picks the machines

No cluster, no executors to size. Dremel fans your query across thousands of slots and bills per TB scanned (or per reserved slot). Best for declarative analytics and ad-hoc SQL.

PySpark side

You write code; you size the cluster

A driver plans, executors crunch partitions, shuffles regroup data. Best for arbitrary Python, ML pipelines, multi-step transforms and custom logic SQL can't express.

The bridge

Same bytes, two engines

Through BigLake + the Storage Read API, both engines read one governed copy. That's the whole lakehouse promise: choose the engine per job, never duplicate the data.

These operations express the same intent and run on the same kind of plan in both engines — a filter is a filter, a GROUP BY is a hash-aggregate. Watch the rows flow through the shared pattern, and compare the two dialects line for line.

▶ Tap any stage above for what it does in each engine. Below: the identical job, both dialects.

BigQuery Standard SQL
SELECT region, SUM(amount) AS rev FROM `proj.gold.fct_orders` WHERE dt = '2026-06-11' AND status = 'paid' GROUP BY region ORDER BY rev DESC LIMIT 10;
PySpark DataFrame
(df .filter((col("dt")=="2026-06-11") & (col("status")=="paid")) .groupBy("region") .agg(sum("amount").alias("rev")) .orderBy(col("rev").desc()) .limit(10))
Identical 01

Filter = predicate pushdown

Both engines push the WHERE / .filter() down to the scan so non-matching rows are never fully read. In the connector this travels server-side via the Storage Read API.

Identical 02

Project = column pruning

SELECT col and .select() both read only the chosen columns from columnar storage. This is why SELECT * / df with no projection is the costliest habit in either engine.

Identical 03

GROUP BY = hash aggregate

Both build a hash aggregate and regroup rows by key — BigQuery in its mixer tree, Spark via a shuffle. Same algorithm, different transport.

Where they diverge. Some jobs are trivial in one engine and painful (or impossible) in the other. Pick the right tool per task.

Pattern 01

Push the heavy reduction to SQL

Billion-row joins and aggregations are what BigQuery slots are built for. Reduce first, then hand a small result to Spark — cheaper and faster than shuffling everything in the cluster.

Pattern 02

Finish with Python in Spark

Model inference, fuzzy matching, nested reshaping, anything with a pip dependency — do it on the already-small result set.

Pattern 03

MERGE / CDC to commit deltas

Spark stages the changed rows; a single MERGE INTO (BigQuery or Iceberg) applies them atomically. Time travel gives you the audit trail for free.

12 · The toolbox

The 7 GCP services, explained like you're new 🧰

Every box you saw in the pipeline is one of these managed services. Tap a card to expand it — what it is, what it's great at, a beginner analogy, and the exact job it does in our lakehouse.

What it is

Google's object storage: you put files (objects) into buckets, and it scales to exabytes with 11 nines of durability. It doesn't run compute — it just holds bytes, extremely well and extremely cheaply.

Key features
  • Storage classes — Standard, Nearline, Coldline, Archive; lifecycle rules auto-move old files to cheaper tiers.
  • Strong consistency — read-after-write, which table formats like Iceberg rely on.
  • Fine-grained IAM + uniform bucket-level access for security.
  • Versioning & retention locks — protect raw data from accidental deletes.
🪣 Analogy: a bottomless, fire-proof warehouse with perfect shelving labels. It never loses a box, but it won't open the boxes for you — that's the compute engines' job.
📍 In our pipeline: hosts all three Medallion zones as buckets/prefixes — raw/ Parquet, conformance/ and curated/ Iceberg files. Every other service reads or writes here.
What it is

A serverless cron service. You define a schedule like 0 2 * * * (2 AM daily) and a target — an HTTP endpoint, a Pub/Sub topic, or a Cloud Run job — and Google guarantees it fires on time.

Key features
  • Standard cron syntax with time-zone awareness (no DST surprises).
  • Retries with backoff if the target fails or times out.
  • Targets: HTTP(S), Pub/Sub, App Engine — perfect for kicking off Cloud Run jobs.
  • Pennies per job per month — no server to babysit.
⏰ Analogy: an alarm clock that never needs batteries and can press one button somewhere in the cloud when it rings.
📍 In our pipeline: rings at 02:00 every night and triggers the Cloud Run ingestion job that pulls fresh source data into the Raw zone.
What it is

Serverless batch execution: package any code as a container, and Cloud Run runs it until it exits. No HTTP server needed, no idle cost — it spins up, does the work, writes results, and disappears.

Key features
  • Run-to-completion semantics with automatic retries on failure.
  • Parallel tasks — fan one job out into N workers (e.g. one per source table).
  • Up to 24h runtime, generous CPU/RAM, any language that fits in a container.
  • Triggered by Scheduler, Workflows, Airflow, or gcloud run jobs execute.
📦 Analogy: a day-labourer you summon by text: shows up, does exactly one task from the checklist, leaves, and only invoices for minutes worked.
📍 In our pipeline: the ingestion job — calls source APIs / reads source DB extracts and lands them as immutable Parquet in the Bronze (Raw) zone.
What it is

The request-driven side of Cloud Run: a container (Service) or a single function (Cloud Run functions) that wakes on an HTTP request or an event — like a file landing in GCS — handles it, and scales back to zero.

Key features
  • Scale 0 → thousands of instances automatically with traffic.
  • Event triggers via Eventarc: GCS object created, Pub/Sub message, audit logs…
